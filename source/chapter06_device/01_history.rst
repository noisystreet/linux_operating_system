======================
设备管理发展史
======================

操作系统不仅要管理 CPU 和内存，还要协调键盘、磁盘、网卡等五花八门的硬件。设备管理技术的演进，体现了如何在"统一抽象"与"硬件多样性"之间寻找平衡。

早期：轮询与专用接口
========================

第一批计算机的外设极少，CPU 通过:strong:`轮询` （polling）不断检查设备是否就绪——简单但浪费 CPU。随后:strong:`中断` （interrupt）出现：设备就绪时向 CPU 发信号，CPU 暂停当前工作处理中断，效率大幅提升。

早期系统为每种设备编写专用代码，与内核紧密耦合。新增设备需修改内核，难以维护和移植。

Unix 设备文件抽象
========================

Unix 将设备抽象为:strong:`设备文件` ，位于 ``/dev`` 下，通过 read/write/ioctl 访问，与普通文件接口一致。分为：

- :strong:`字符设备` （char device）：按字节流访问，如键盘、串口、``/dev/null``
- :strong:`块设备` （block device）：按块随机访问，如磁盘，有缓存和调度

这一抽象使应用程序无需关心硬件细节，驱动只需实现标准接口。

Linux 2.4 与 Devfs
========================

Linux 2.4 引入:strong:`devfs` ，在内核中动态创建 ``/dev`` 节点，替代静态的 ``/dev/MAKEDEV`` 脚本。但 devfs 存在设计缺陷，维护困难，后被移除。

可加载模块与 Plug and Play
==================================

:strong:`可加载内核模块` （LKM）使驱动可编译为 ``.ko`` 文件，运行时 ``insmod`` 加载、``rmmod`` 卸载，无需重编译整个内核。硬件:strong:`即插即用` （PnP）和:strong:`热插拔` 要求系统能动态识别新设备并加载对应驱动。

udev 与设备模型
========================

2003 年起，:strong:`udev` 接管用户空间的设备节点管理。内核通过 sysfs 导出设备信息，udev 根据规则创建 ``/dev`` 节点、设置权限、创建符号链接（如 ``/dev/disk/by-uuid/``）。devfs 在 2.6.13 后移除，udev 成为标准。

统一驱动模型
========================

Linux 2.6 引入:strong:`统一设备模型` （Unified Driver Model）：用 bus、device、driver 三层结构描述硬件拓扑和驱动绑定关系，sysfs 呈现为 ``/sys`` 目录树。PCI、USB、platform 等总线均纳入该模型。

现代 I/O 栈
========================

存储 I/O 路径持续演进：

.. list-table::
   :header-rows: 1
   :widths: 18 42

   * - 阶段
     - 发展
   * - PIO
     - CPU 逐字节搬运，极慢
   * - DMA
     - 设备直接访问内存，CPU 解放
   * - 中断合并
     - 减少中断风暴
   * - 多队列
     - NVMe、virtio-blk 每 CPU 独立队列
   * - io_uring
     - 异步块 I/O 与用户态高效交互

NVMe SSD 取代机械硬盘成为主流，I/O 调度从"减少寻道"转向"降低延迟、提高并行度"。

以一块 NVMe 盘为例，从应用到硅片的完整路径可概括为：

.. code-block:: text

   read() / write()
   → VFS → 文件系统（ext4 等）→ page cache
   → 块层（blk-mq 多队列）
   → NVMe 驱动 → PCIe → SSD 控制器

理解这条链路有助于将第 5 章文件 I/O 与第 6 章块设备、中断、DMA 联系起来。``iostat -x`` 中的 ``%util``、``await`` 反映的是块层及以下瓶颈，而非 VFS 本身。
