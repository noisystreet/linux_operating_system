======================
ext4 文件系统
======================

ext4 是 ext2/ext3 的继任者，也是多数 Linux 发行版的默认根文件系统。它在保持向后兼容的同时，引入了 extents、延迟分配、更大容量等特性。本节介绍 ext4 的磁盘布局、inode 结构和日志机制。

ext 家族简史
========================

.. list-table::
   :header-rows: 1
   :widths: 12 18 50

   * - 版本
     - 年代
     - 主要特性
   * - ext2
     - 1993
     - 经典 Unix FS 布局，无日志
   * - ext3
     - 2001
     - 增加 journal，在线扩容
   * - ext4
     - 2008
     - extents、延迟分配、更大文件/FS

ext4 挂载为 ext3 或 ext2 时，部分新特性不可用，但数据可读。

磁盘布局
========================

ext4 将分区划分为:strong:`块组` （block group），每个块组包含：

.. code-block:: text

   块组 N
   ┌────────────────┐
   │ 超级块副本      │  （部分块组有备份）
   ├────────────────┤
   │ 块组描述符      │
   ├────────────────┤
   │ 块位图          │  哪些数据块已用
   ├────────────────┤
   │ inode 位图      │  哪些 inode 已用
   ├────────────────┤
   │ inode 表        │  多个 inode 结构
   ├────────────────┤
   │ 数据块          │  文件内容和目录项
   └────────────────┘

:strong:`超级块` 描述整个文件系统：块大小、块总数、inode 总数、UUID 等。挂载时首先读取超级块验证魔数（``0xEF53``）。

inode 与 extents
========================

ext2/3 用:strong:`直接块 + 一级/二级/三级间接块` 指针定位数据，大文件需多次间接访问。ext4 引入:strong:`extents` ：连续块可用一个 extent 描述（起始块 + 长度），减少元数据、提高大文件效率。

inode 结构（简化）包含：mode、uid、gid、大小、时间戳、块指针或 extent 树。目录文件的"数据块"存放 ``{inode, name_len, file_type, name}`` 格式的目录项。

.. code-block:: bash

   sudo dumpe2fs /dev/sda2 | head -50   # 查看超级块和块组信息
   sudo debugfs -R "stat <inode>" /dev/sda2   # 调试 ext 文件系统

日志（Journal）
========================

ext4 支持三种日志模式（挂载选项 ``data=``）：

.. list-table::
   :header-rows: 1
   :widths: 15 45

   * - 模式
     - 行为
   * - ordered
     - 默认。元数据日志，数据先写盘再提交元数据
   * - writeback
     - 元数据日志，数据写入顺序不保证，最快但风险较高
   * - journal
     - 元数据与数据都写日志，最安全，最慢

日志是环形的专用区域或外部文件（``journal`` inode）。崩溃后 ``fsck`` 或自动重放使元数据一致。

延迟分配与多块分配
========================

:strong:`延迟分配` （delayed allocation）：写入 page cache 时不立即分配磁盘块，直到 flush 或内存压力时再分配。好处：

- 合并连续写入，减少碎片
- 短生命周期文件可能从未落盘，直接丢弃

:strong:`多块分配器` （mballoc）尝试为相关文件分配相邻块，提升顺序读写性能。

用 dumpe2fs 观察 on-disk 结构
========================

``dumpe2fs`` 读取超级块和块组描述符，是理解 ext4 布局的首选工具（对未挂载设备操作，避免破坏数据）：

.. code-block:: bash

   sudo dumpe2fs -h /dev/sda2
   # 关注：Block size, Block count, Inode count, Journal inode

输出中的 ``Filesystem features`` 列出已启用特性，如 ``has_journal``、``extent``、``64bit``。``First block`` 和 ``Block group`` 段展示各块组的空闲块/inode 数量。

extent 树示意
========================

ext4 大文件用 extent 树替代间接块链。每个 extent 记录 ``{逻辑块起始, 物理块起始, 长度}`` ，一段连续 1 MB 数据只需一个 extent 项，而 ext2 间接块方案可能需要数百个指针。

.. code-block:: text

   inode
     └── extent 头
           ├── extent[0]: logical 0, physical 1000, len 256 blocks
           ├── extent[1]: logical 256, physical 5000, len 128 blocks
           └── ...

``debugfs`` 可查看具体文件的 extent 映射：

.. code-block:: bash

   sudo debugfs -R "dump_extents /etc/hostname" /dev/sda2

崩溃恢复与 fsck
========================

异常断电后，内核挂载时检查日志是否需要重放。若超级块或日志严重损坏，需离线修复：

.. code-block:: bash

   sudo fsck.ext4 -f /dev/sda2

``-f`` 强制检查即使文件系统标记为 clean。生产环境应先备份或确认维护窗口——fsck 对大分区可能耗时数小时。``tune2fs -l`` 可查看上次检查时间和挂载计数。
