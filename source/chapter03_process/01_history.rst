======================
进程管理发展史
======================

操作系统之所以能让多个程序"同时"运行，核心在于 **进程** （process）这一抽象。进程概念的演进，贯穿了操作系统从批处理到分时、从单核到多核的整个历史。理解这段历史，有助于把握 Linux 进程管理的设计取舍。

从作业到进程
========================

早期批处理系统中，内存里一次只驻留一个"作业"（job）。作业运行到结束才释放全部资源，没有中途切换的概念。操作员提交的作业队列由操作系统依次调度执行——这可以看作进程模型的雏形，但还缺少隔离和保护。

1960 年代 **多道程序** （multiprogramming）出现后，多个程序同时驻留内存。操作系统需要在它们之间切换 CPU，并防止一个程序破坏另一个程序的内存。**进程** 作为"运行中的程序实例"这一概念由此确立——它不仅是代码，还包括当前执行状态、占用的资源和地址空间。

分时系统要求切换更快、更频繁。MIT 的 CTSS 和后来的 Multics 将进程切换时间压缩到毫秒级，用户感觉自己在独占机器。Unix 继承并简化了这套模型，进程成为 Unix 操作系统最核心的抽象之一。

Unix 进程模型的奠基
========================

Unix（1969）对进程管理的贡献影响深远：

- **fork()** ：通过复制父进程创建子进程，简洁而强大
- **exec()** 族 ：用新程序替换当前进程的映像，fork 与 exec 分离
- **wait()** ：父进程等待子进程退出并回收资源
- **PID** ：每个进程有唯一标识符，init 收养孤儿进程

这套 API 沿用至今。Linux 作为 Unix 的后继者，完整继承了 fork/exec/wait 语义，并在实现上做了大量优化。

多任务与抢占式调度
========================

单任务时代，程序运行到结束才释放 CPU。**协作式多任务** （cooperative multitasking）时代，程序主动让出 CPU（如 Windows 3.x 的 ``Yield``）。一个程序死循环就能拖垮整个系统。

**抢占式多任务** （preemptive multitasking）由操作系统强制切换——每个进程运行一段时间后，时钟中断触发调度器选择下一个进程。Unix 从早期就采用抢占式调度，Linux 同样如此。这是桌面系统保持响应性的基础。

线程概念的引入
========================

传统进程是 **资源分配** 的单位，切换开销较大（需要切换页表、刷新 TLB）。1980 年代，研究者提出 **线程** （thread）——在同一进程内共享地址空间的多个执行流，切换时无需更换页表。

线程模型经历了用户级线程（如 GNU Pth）、内核级线程（如 Windows NT）到 **混合模型** 的演进。Linux 在 2.6 版本后采用 NPTL（Native POSIX Thread Library），将线程实现为共享地址空间的轻量级进程，与内核调度器深度集成。

调度算法的迭代
========================

随着进程数量增长，调度算法也在不断进化：

.. list-table::
   :header-rows: 1
   :widths: 18 22 40

   * - 年代
     - 算法
     - 特点
   * - 1960s
     - 轮转（RR）
     - 时间片轮转，简单公平
   * - 1970s
     - 多级反馈队列
     - 根据行为动态调整优先级
   * - 1990s
     - O(n) 调度器
     - Linux 2.4 及以前，随进程数线性变慢
   * - 2003
     - O(1) 调度器
     - Linux 2.6，常数时间选择下一进程
   * - 2007
     - CFS
     - Linux 2.6.23+，基于虚拟运行时间的完全公平调度
   * - 持续
     - 实时调度
     - SCHED_FIFO、SCHED_RR 满足硬实时需求

今天 Linux 桌面和服务器默认使用 CFS，实时任务使用 SCHED_FIFO/SCHED_RR，节能场景还有 SCHED_IDLE 等策略。

多核时代的挑战
========================

单核时代，调度器只需决定"下一个运行谁"。多核时代增加了新问题：

- **负载均衡** ：如何避免某个 CPU 过载而其他 CPU 空闲
- **CPU 亲和性** ：线程是否绑定特定核心，利用缓存局部性
- **NUMA** ：非统一内存访问架构下，进程应优先使用本地内存节点

Linux 调度器（``kernel/sched/core.c``、``kernel/sched/fair.c``）持续演进，支持 per-CPU 运行队列、组调度（cgroup）、带宽控制等特性。容器和 cgroup 的流行，更让"进程调度"与"资源配额"紧密耦合。

从进程到协程
========================

近年 **协程** （coroutine）和 **异步 I/O** 在应用层流行（如 Go goroutine、C++20 coroutine）。它们通常构建在线程之上，由用户态调度，减少内核切换开销。但内核眼中的执行单元仍是线程——协程是语言运行时对线程的复用和包装。

无论上层模型如何变化，操作系统内核提供的进程与线程抽象仍是基石。下一节从进程的基本概念和 PCB 结构入手，进入 Linux 进程管理的核心。
