======================
虚拟内存
======================

每个进程都以为自己拥有从 0 到最大地址的连续空间，实际上这些地址是:strong:`虚拟` 的，由操作系统和 MMU 映射到物理内存或磁盘。本节介绍虚拟内存的动机、请求调页、写时复制和 ``mmap`` 系统调用。

为什么需要虚拟内存
========================

没有虚拟内存时，程序直接使用物理地址，带来诸多问题：

- 多个程序无法安全共存，需手动规划加载地址
- 程序大小受物理内存限制
- 内存碎片导致无法加载大程序
- 无法灵活共享库或只读代码

:strong:`虚拟内存` 为每个进程提供独立的:strong:`虚拟地址空间` ，操作系统通过页表将虚拟页映射到物理页框或磁盘上的交换区。进程只看到自己的地址空间，无法访问其他进程或内核（除非通过系统调用）。

请求调页
========================

:strong:`请求调页` （demand paging）是虚拟内存的核心策略：进程启动时并不立即加载全部代码和数据，而是当首次访问某页时，若该页不在内存，触发:strong:`缺页异常` （page fault）。

缺页异常处理流程（简化）：

1. CPU 访问虚拟地址，MMU 发现页表项 Present=0 或访问权限不符
2. 触发 page fault，陷入内核
3. 内核的 ``do_page_fault()`` （``arch/x86/mm/fault.c``）分析原因
4. 若为合法访问：从磁盘换入该页（或分配零页），更新页表，返回用户态重试
5. 若为非法访问（如空指针、越界）：发送 ``SIGSEGV`` 给进程

这种"用时再加载"策略使启动大量进程时内存占用极低，只有实际访问的页才占用物理内存。

写时复制（COW）
========================

:strong:`写时复制` （Copy-On-Write）延迟内存复制，直到真正写入时才复制物理页。典型场景：

:strong:`fork()`
  子进程复制父进程页表，父子共享同一物理页，页表项标记为只读。任一方写入时触发缺页，内核复制该页，更新页表，使写入方获得独立副本。

:strong:`mmap(MAP_PRIVATE)`
  私有映射的修改不反映到文件或其他映射，通过 COW 实现——多个进程映射同一文件时共享只读页，写入时各自复制。

COW 使 ``fork()`` 对大进程非常高效，也是 Docker 等容器技术的基础之一。

mmap 系统调用
========================

``mmap()`` 将文件或匿名内存映射到进程地址空间：

.. code-block:: cpp

   void* addr = mmap(nullptr, length, prot, flags, fd, offset);

常用参数：

.. list-table::
   :header-rows: 1
   :widths: 22 48

   * - 参数
     - 说明
   * - ``prot``
     - PROT_READ | PROT_WRITE | PROT_EXEC
   * - ``flags``
     - MAP_SHARED（共享）/ MAP_PRIVATE（私有 COW）
   * - ``MAP_ANONYMOUS``
     - 匿名映射，不关联文件，常用于堆扩展
   * - ``fd, offset``
     - 文件描述符和偏移，匿名时 fd=-1

``mmap`` 相比 ``malloc`` 的优势：可映射大文件、进程间共享、减少数据拷贝。数据库、视频处理等常用 ``mmap`` 访问大文件。

释放映射用 ``munmap(addr, length)``。

内存映射的布局
========================

典型 Linux 进程虚拟地址空间布局（x86-64）：

.. code-block:: text

   高地址
   ┌─────────────────┐
   │  内核空间        │  （用户态不可见）
   ├─────────────────┤
   │  栈 (stack)      │  向下增长
   │       ↓          │
   │  ... 空闲 ...    │
   │       ↑          │
   │  堆 (heap)       │  向上增长（brk/mmap）
   ├─────────────────┤
   │  BSS / 数据段    │
   ├─────────────────┤
   │  代码段 (text)   │  只读、可执行
   └─────────────────┘
   低地址 0x400000 附近

可用 ``cat /proc/self/maps`` 查看实际布局。栈和堆之间的区域是"空洞"，访问会触发 ``SIGSEGV``。

缺页与 SIGSEGV
========================

并非所有缺页都是"换页"——访问未映射区域、违反权限（如写只读页、执行数据页）会触发:strong:`段错误` （SIGSEGV）。这是内存保护机制在起作用。

.. code-block:: cpp

   int* p = nullptr;
   *p = 42;   // SIGSEGV

调试段错误可用 ``gdb`` 查看崩溃时的栈和地址，或分析 core dump（``ulimit -c unlimited`` 启用）。

过度提交与 OOM
========================

Linux 允许:strong:`过度提交` （overcommit）——分配的虚拟内存总和可超过物理内存，因为并非所有页都会立即使用。当物理内存耗尽，内核的:strong:`OOM Killer` （Out-Of-Memory Killer）会选择一个进程杀死以释放内存，通常选择占用大、优先级低的进程。

.. code-block:: bash

   # 查看 OOM 分数（越高越可能被杀）
   cat /proc/self/oom_score
