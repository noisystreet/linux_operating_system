======================
内存管理发展史
======================

内存是程序运行的舞台。从早期程序员直接操作物理地址，到现代虚拟内存抽象，内存管理技术的演进决定了操作系统能支持多大的程序、多强的隔离、多高的利用率。本节回顾内存管理的关键里程碑。

早期：直接访问物理地址
========================

第一代计算机没有内存管理硬件。程序员在汇编中直接使用物理地址，程序必须加载到固定内存位置。多个程序无法安全共存——一个程序的 bug 可能覆盖另一个程序或操作系统本身。

:strong:`基址寄存器` （base register）和:strong:`界限寄存器` （limit register）的出现，使 CPU 硬件检查每次内存访问是否在合法范围内，为多道程序提供了最基本的保护。程序使用逻辑地址，硬件自动加上基址得到物理地址。

分段时代
========================

:strong:`分段` （segmentation）按逻辑单位（代码段、数据段、栈段）划分地址空间。每个段有独立的基址和界限，段内偏移由程序指定。Intel 8086 的段:偏移寻址是典型代表。

分段的问题：段大小可变，物理内存容易产生:strong:`外部碎片` —— 总空闲足够，但没有连续大块满足新段需求。段的大小也缺乏灵活性，共享代码段、保护粒度等设计复杂。

分页的兴起
========================

:strong:`分页` （paging）将物理内存和虚拟地址空间划分为固定大小的:strong:`页` （page），通常 4 KB。虚拟地址通过:strong:`页表` 映射到物理页框（frame）。碎片问题转化为:strong:`内部碎片` —— 进程最后一页可能未用满，但远小于分段的外部碎片。

1970 年代，分页与分段结合（如 x86）或纯分页（如 DEC VAX）成为主流。页是内存分配、换入换出的基本单位，为虚拟内存奠定基础。

虚拟内存
========================

:strong:`虚拟内存` 让每个进程拥有独立的虚拟地址空间，大小可远超物理内存。操作系统将暂时不用的页换出到磁盘，需要时再换入——:strong:`请求调页` （demand paging）。

1960 年代 Atlas 计算机首次实现虚拟内存。1970 年代，Multics 和 Unix 将虚拟内存带入通用系统。虚拟内存带来：

- :strong:`隔离` ：进程无法访问彼此地址空间
- :strong:`超用` ：物理内存可小于所有进程需求之和
- :strong:`简化链接` ：每个程序可从固定虚拟地址（如 0）开始链接

高速缓存与 TLB
========================

页表存放在内存中，每次访存需两次内存访问（查页表 + 访问数据）。:strong:`TLB` （Translation Lookaside Buffer）是页表项的硬件缓存，命中时一次访问即可。TLB 未命中（page walk）需遍历多级页表，代价较高。

x86-64 采用:strong:`四级页表` ，ARM64 采用多级页表，现代 CPU 的 TLB 还支持与进程绑定的 ASID（Address Space ID），减少进程切换时的 TLB 刷新。

物理内存分配算法
========================

内核需要管理物理页框的分配与回收。经典算法：

.. list-table::
   :header-rows: 1
   :widths: 20 50

   * - 算法
     - 特点
   * - 位图
     - 每位表示一页，简单但分配慢
   * - 空闲链表
     - 按地址排序，易合并碎片
   * - 伙伴系统
     - Linux 采用，按 2 的幂次分配，快速合并
   * - slab
     - 缓存小对象，减少碎片和初始化开销

Linux 的:strong:`伙伴系统` （buddy system）和:strong:`slab 分配器` 自 2.0 时代演进至今，slab 后被 SLUB（默认）和 SLOB（嵌入式）补充。

大内存与 NUMA
========================

单机内存从 MB 到 GB 再到 TB，:strong:`NUMA` （Non-Uniform Memory Access）架构成为主流——多 CPU 插座，每插座有本地内存，访问远程内存延迟更高。操作系统需将进程和内存分配在"就近"的 NUMA 节点，Linux 的 NUMA 策略和 ``numactl`` 即为此服务。

持久内存、CXL 等新技术正在模糊内存与存储的边界，但虚拟内存 + 分页的核心模型在可预见的未来仍将主导。

过度提交与 OOM
========================

Linux 默认允许:strong:`内存过度提交` （overcommit）：``fork`` 时仅复制页表不立即分配物理页，许多进程「承诺」的虚拟内存总和可超过物理 RAM。这提高 fork 效率，但在极端情况下所有进程同时触碰内存时会触发:strong:`OOM killer` 。

内核按 ``oom_score`` 选择牺牲进程（通常高内存、低优先级用户进程）。容器场景下 cgroup ``memory.max`` 可在组内 OOM，避免拖垮整台主机——详见第 10 章 cgroup 节。

从分段分页到虚拟内存，从伙伴系统到 NUMA，内存管理为进程提供了安全、灵活的地址空间。下一节从硬件寻址机制入手，理解 x86-64 如何将虚拟地址翻译成物理地址。
