======================
启动技术发展史
======================

按下电源键，屏幕亮起，操作系统出现——这个过程我们每天都在经历，却很少去想中间发生了什么。要理解现代 Linux 的启动流程，不妨先回顾启动技术本身是如何演进的。每一次固件标准的换代，都深刻改变了操作系统与硬件之间的交接方式。

从跳线到固件
========================

最早的计算机没有"启动程序"的概念。操作员通过面板开关设定起始地址，按下"运行"按钮，CPU 就从那个地址开始取指令。程序和数据通过穿孔卡片或纸带输入，每次开机都是一次"冷启动"。

1970 年代，随着 ROM 芯片成本下降，:strong:`固件` （firmware）出现了——一段固化在主板芯片上的程序，上电后 CPU 自动从这里开始执行。IBM PC（1981）采用的 :strong:`BIOS` （Basic Input/Output System）成为个人电脑时代的标准，这一标准统治了将近三十年。

BIOS 时代的局限
========================

BIOS 的设计诞生于 16 位实模式时代，其架构在 21 世纪逐渐显得力不从心：

- :strong:`寻址限制` ：传统 BIOS 运行在实模式，只能直接访问 1 MB 以下内存，加载大体积内核需要复杂的模式切换
- :strong:`启动速度慢` ：BIOS 逐一初始化硬件，缺乏并行能力；POST（加电自检）过程冗长
- :strong:`安全性薄弱` ：没有统一的启动链验证机制，引导扇区病毒曾是常见威胁
- :strong:`分区限制` ：MBR 分区表最多支持 2 TB 磁盘和 4 个主分区

尽管如此，BIOS 的接口约定（中断向量 ``INT 13h`` 读磁盘、``INT 10h`` 显示字符）被无数引导加载程序依赖，形成了庞大的生态惯性。

UEFI 的登场
========================

2005 年前后，:strong:`UEFI` （Unified Extensible Firmware Interface）开始取代传统 BIOS。UEFI 并非简单的"新 BIOS"，而是一套完整的固件规范，由 UEFI 论坛维护。

UEFI 带来的关键变化：

- :strong:`原生 64 位` ：固件本身可以运行在 64 位长模式，直接加载现代内核
- :strong:`GPT 分区` ：支持超过 2 TB 的磁盘和 128 个主分区
- :strong:`Secure Boot` ：通过数字签名验证引导加载程序和内核，防止恶意代码在启动阶段植入
- :strong:`驱动模型` ：UEFI 内置网络、文件系统等驱动，固件阶段就能访问 ext4、FAT 分区

今天购买的新电脑几乎全部采用 UEFI。Linux 发行版也普遍以 UEFI 模式安装，BIOS 模式（Legacy Boot）正在退出历史舞台。

引导加载程序的演进
========================

固件只负责完成硬件初始化和找到可引导设备，真正加载操作系统内核的任务落在 :strong:`引导加载程序` （bootloader）肩上。

.. list-table::
   :header-rows: 1
   :widths: 15 20 45

   * - 时代
     - 代表
     - 特点
   * - 1980s
     - MS-DOS 引导扇区
     - 512 字节的 MBR 引导代码，只能加载相邻扇区
   * - 1990s
     - LILO
     - Linux Loader，配置简单但修改内核后需重新运行 lilo 命令
   * - 2000s
     - GRUB Legacy
     - 支持多操作系统、文件系统级引导，成为 Linux 标配
   * - 2010s
     - GRUB 2
     - 模块化架构，支持 Btrfs/ZFS 子卷、图形菜单、脚本化配置
   * - 2010s
     - systemd-boot
     - 轻量级 UEFI 引导器，仅支持 UEFI + GPT，配置极简

GRUB 2 是目前 Linux 世界最主流的引导加载程序。systemd-boot（旧名 gummiboot）则在 Arch Linux、部分 Fedora 安装中作为 UEFI 环境下的轻量替代方案。

init 系统的变迁
========================

内核启动完成后，需要启动第一个用户态进程——:strong:`init` （PID 1）。init 负责挂载根文件系统、启动系统服务、回收孤儿进程。init 系统的演进反映了操作系统设计理念的变化：

- :strong:`SysVinit` （1980s–2010s）：基于运行级别（runlevel）和 Shell 脚本，顺序启动服务，逻辑清晰但启动慢
- :strong:`Upstart` （2006–2014）：Ubuntu 采用，引入事件驱动，部分服务可并行启动
- :strong:`systemd` （2010–至今）：集 init、服务管理、日志、设备管理于一体，通过依赖关系图实现大规模并行启动

今天的主流 Linux 发行版——Ubuntu、Fedora、Debian、RHEL——均已采用 systemd 作为 init 系统。理解启动流程，就必须理解 systemd 在其中的角色。

从加电到 Shell
========================

把上述演进串成一条时间线，现代 PC 的启动路径大致如下：

.. code-block:: text

   加电 → UEFI 固件（POST、硬件初始化）
        → UEFI 加载 EFI 分区中的引导加载程序（如 GRUB）
        → GRUB 加载内核镜像（vmlinuz）和 initramfs
        → 内核解压、初始化子系统、挂载根文件系统
        → 内核启动 init 进程（systemd）
        → systemd 按依赖顺序启动各服务
        → 图形界面或登录提示符出现

前沿方向：:strong:`UKI` （Unified Kernel Image）将内核、initramfs、内核命令行打包为单个 UEFI 可执行文件，简化 Secure Boot 签名与部署；systemd-boot 与部分发行版已支持。传统 ``vmlinuz`` + ``initrd.img`` 分离模式仍将长期并存，排障时须区分二者（见本章 outlook 与 ``04_linux_boot``）。
