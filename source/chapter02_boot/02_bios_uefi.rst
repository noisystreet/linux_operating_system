======================
BIOS 与 UEFI
======================

固件是启动链的第一环。你按下电源键后，CPU 执行的最初几条指令并非来自硬盘上的操作系统，而是来自主板 ROM 中的固件程序。本节对比 BIOS 与 UEFI 两种固件标准，理解它们如何完成硬件初始化并把控制权交给引导加载程序。

加电后第一件事：POST
========================

无论是 BIOS 还是 UEFI，上电后的第一步都是 **POST** （Power-On Self-Test，加电自检）。固件会检查：

- CPU 是否正常工作
- 内存是否可读写（UEFI 还会检测每条 DIMM 的容量和频率）
- 键盘、显示器等关键外设是否响应
- 存储控制器和 PCIe 设备是否枚举成功

如果 POST 发现致命错误，固件会通过蜂鸣码（beep code）或屏幕报错提示，启动流程中止。通过 POST 后，固件开始初始化芯片组、配置内存控制器、枚举 PCI 设备。

BIOS 的工作方式
========================

传统 BIOS 运行在 x86 的 **实模式** （real mode）下——16 位寻址，可直接访问的内存上限为 1 MB（其中 640 KB 常规内存，384 KB 保留给显存和 ROM）。

BIOS 的启动流程：

1. CPU 从地址 ``0xFFFF0`` 开始执行（复位向量，位于 BIOS ROM 映射区域）
2. BIOS 完成 POST 和硬件初始化
3. 按照 CMOS 中设定的启动顺序，依次尝试从硬盘、光驱、USB、网络启动
4. 读取目标设备的 **引导扇区** （boot sector）——对于 MBR 磁盘，即第 0 号扇区的前 512 字节
5. 将这 512 字节加载到内存 ``0x7C00``，跳转执行

.. note::

   MBR 引导扇区的 512 字节中，最后 2 字节必须是引导签名 ``0x55 0xAA``。空间极其有限，因此 MBR 引导代码通常只做一件事：加载下一阶段的引导程序（如 GRUB 的 core.img）。

BIOS 通过 **中断向量** 为上层软件提供硬件访问接口。引导加载程序在实模式下运行，通过 ``INT 13h`` 读取磁盘扇区、``INT 10h`` 在屏幕上输出文字。这些接口简单但功能有限，也是 BIOS 被诟病"陈旧"的原因之一。

UEFI 的工作方式
========================

UEFI 与 BIOS 的根本区别在于：UEFI 是一套完整的 **固件操作系统**，而不仅仅是硬件初始化代码。

UEFI 启动流程：

1. CPU 从 Flash 中的 UEFI 固件入口开始执行（**SEC** 阶段）
2. 初始化内存和基础服务（**PEI** 阶段）
3. 加载 UEFI 驱动，枚举所有硬件（**DXE** 阶段）
4. 进入 **BDS** （Boot Device Selection）阶段——这是用户最常感知的部分：显示启动菜单或按启动顺序选择设备
5. 加载 EFI 分区（ESP）中的 ``.efi`` 可执行文件（如 ``grubx64.efi``）
6. 将控制权交给引导加载程序

UEFI 原生支持 32 位和 64 位模式，可以直接访问全部物理内存。它内置了 FAT 文件系统驱动，因此引导加载程序以标准文件形式存放在 EFI 系统分区中，不再依赖隐藏的引导扇区。

EFI 系统分区
========================

UEFI 模式下，磁盘上需要一个特殊的 **EFI 系统分区** （ESP，EFI System Partition）：

- 分区类型 GUID 为 ``C12A7328-F81F-11D2-BA4B-00A0C93EC93B``
- 通常格式化为 FAT32，大小 100–550 MB
- 存放引导加载程序的 ``.efi`` 文件和配置

典型的 ESP 目录结构：

.. code-block:: text

   /EFI/
   ├── BOOT/
   │   └── BOOTX64.EFI        # UEFI 默认引导路径（Fallback）
   ├── grub/
   │   └── grubx64.efi        # GRUB 引导程序
   └── systemd/
       └── systemd-bootx64.efi

Linux 下 ESP 通常挂载在 ``/boot/efi``。可以用以下命令查看：

.. code-block:: bash

   findmnt /boot/efi
   ls /boot/efi/EFI/

MBR 与 GPT 分区表
========================

分区表格式与固件模式密切相关：

.. list-table::
   :header-rows: 1
   :widths: 15 20 45

   * - 格式
     - 配合固件
     - 限制
   * - MBR
     - BIOS（也可用于 UEFI）
     - 最多 4 个主分区，单盘最大 2 TB
   * - GPT
     - UEFI（推荐）
     - 最多 128 个分区，支持 18 EB 磁盘

GPT 在磁盘首尾各保存一份分区表副本，具备 CRC 校验，比 MBR 可靠得多。现代 Linux 安装程序默认使用 GPT + UEFI 组合。

.. warning::

   在 UEFI 模式下用 MBR 分区表安装系统虽然可行（称为 UEFI/GPT 或 UEFI/MBR 混合模式），但并非标准做法，可能在某些主板上引发兼容问题。新安装建议始终使用 GPT。

Secure Boot
========================

UEFI 引入的 **Secure Boot** 是一项安全机制：固件只加载经过签名验证的引导加载程序和内核，防止引导扇区病毒和 rootkit 在操作系统加载前植入。

工作流程：

1. 主板厂商将"平台密钥"（PK）和"密钥交换密钥"（KEK）烧录到固件中
2. 操作系统发行版用微软 UEFI CA 或自有 CA 对引导加载程序和内核签名
3. 启动时，固件验证签名链：PK → KEK → 引导程序签名 → 内核签名
4. 任何一环验证失败，启动被阻止

Linux 发行版普遍支持 Secure Boot：GRUB 和内核都带有微软 UEFI CA 的签名。如果你在安装时启用了 Secure Boot，无需额外操作。自行编译内核则需要手动签名或使用 shim 引导器。

.. hint::

   在虚拟机（如 VMware、VirtualBox、QEMU/KVM）中，Secure Boot 通常默认关闭。物理机上可在 UEFI 设置（按 F2/Del 进入）中开关此选项。

BIOS 与 UEFI 对比
========================

.. list-table::
   :header-rows: 1
   :widths: 20 35 35

   * - 维度
     - BIOS
     - UEFI
   * - CPU 模式
     - 16 位实模式
     - 32/64 位保护/长模式
   * - 磁盘访问
     - INT 13h 按扇区读写
     - 文件系统级访问（FAT 等）
   * - 分区表
     - MBR 为主
     - GPT 为主
   * - 引导程序存放
     - 引导扇区（512 字节）
     - ESP 中的 .efi 文件
   * - 安全启动
     - 不支持
     - Secure Boot
   * - 配置界面
     - 文本菜单（蓝底灰字）
     - 图形化 GUI（部分主板）
