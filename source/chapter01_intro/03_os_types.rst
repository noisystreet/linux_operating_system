======================
操作系统分类
======================

操作系统不是只有 Linux 和 Windows。根据设计目标和应用场景的不同，操作系统在架构和特性上差异巨大。这一节从几个维度对操作系统进行分类，帮助你建立更完整的认知。

按应用场景分类
================

桌面操作系统
--------------

面向个人用户的交互式计算。典型代表有 Windows、macOS、Linux（桌面发行版）。这类操作系统对交互体验和生态丰富度要求高，对实时性要求不高。

服务器操作系统
--------------

面向数据中心和网络服务。Linux 在这个领域占据绝对主导，Windows Server 也有一定份额。这类系统强调：

- :strong:`稳定性` ：无图形界面，可以连续运行数年不重启
- :strong:`安全性与隔离` ：多用户权限管理、SELinux 等
- :strong:`网络性能` ：高效处理大量的并发连接

嵌入式操作系统
--------------

运行在资源受限的设备上。小到智能手表、路由器，大到工业控制器。常见的有：

- :strong:`Linux` （裁剪后）：Android、OpenWrt、Yocto 项目
- :strong:`FreeRTOS`、:strong:`Zephyr`、*RT-Thread*：针对微控制器的实时操作系统

嵌入式系统的核心约束是功耗、内存和计算能力——有时只有几十 KB 的 RAM。

实时操作系统
--------------

上面提到的系统追求的是"平均性能"，而实时操作系统（RTOS）追求的是:strong:`确定性` （determinism）：每个任务的响应时间必须在可预测的范围内。

实时系统分为两类：

- 硬实时（hard real-time）：错过截止时间等于灾难——比如汽车安全气囊必须在碰撞后的几毫秒内弹出
- 软实时（soft real-time）：偶尔错过截止时间可以接受，但会影响体验——比如视频播放掉帧

Linux 本身是通用操作系统，不保证实时性。但通过 :strong:`PREEMPT_RT` 补丁，可以将 Linux 改造为接近实时系统的行为。这也是 Linux 能进入工业控制领域的原因。

按架构分类
================

宏内核（Monolithic Kernel）
---------------------------

宏内核将进程管理、内存管理、文件系统、设备驱动等所有核心功能都运行在同一个内核地址空间中。Linux 和 Unix 是宏内核的典型代表。

优点：性能好——模块之间的通信就是函数调用，没有上下文切换开销。
缺点：内核庞大——一个驱动的 bug 可能导致整个系统崩溃。

微内核（Microkernel）
-----------------------

微内核的理念是:strong:`把最少的功能留在内核态`，只保留进程间通信（IPC）和基本的调度、内存管理。文件系统、设备驱动等作为用户态服务运行。

QNX、MINIX 是微内核的代表。L4 系列微内核广泛应用于手机基带等安全关键领域。

优点：隔离性好——驱动崩溃了不会影响内核，重启驱动即可。
缺点：性能开销——用户态服务之间的通信需要经过 IPC，有上下文切换代价。

宏内核 vs 微内核架构对比
==========================

.. code-block:: text

   宏内核（Linux）                    微内核（MINIX / QNX）
   ┌─────────────────────┐           ┌─────────────────────┐
   │  用户态应用程序      │           │  用户态应用程序      │
   └──────────┬──────────┘           └──────────┬──────────┘
              │ syscall                        │ IPC
   ┌──────────▼──────────────────┐   ┌─────────▼─────────┐
   │  内核态：调度/内存/VFS/驱动  │   │ 微内核：调度/IPC  │
   │  （同一地址空间，函数调用）   │   └─────────┬─────────┘
   └─────────────────────────────┘             │ IPC
                                    ┌──────────▼──────────┐
                                    │ 用户态服务：FS/驱动  │
                                    └─────────────────────┘

Linux 通过:strong:`可加载内核模块` （``insmod``）在宏内核架构下获得一定灵活性——驱动可动态插入，而无需把整个文件系统迁到用户态。

混合内核
--------

现实世界的操作系统往往不那么纯粹。Windows NT 和 macOS XNU 内核属于:strong:`混合内核`——核心是微内核架构，但将部分性能敏感的服务（如文件系统缓存、图形驱动）放回内核态。

外核（Exokernel）
-----------------

外核将硬件资源的管理权直接交给应用程序，内核只负责资源保护和多路复用（multiplexing）。应用程序通过库操作系统（libOS）管理自己申请的资源。这种架构在高性能计算和某些虚拟化场景中有研究价值。

按使用模式分类
================

单用户 vs 多用户
-----------------

- :strong:`单用户` ：MS-DOS、早期 Mac OS
- :strong:`多用户` ：Linux、Unix、Windows NT 系列

多用户意味着系统必须提供完善的权限隔离和资源配额机制，否则一个用户就能侵占全部资源、干扰其他用户的进程。Linux 的权限模型正是为此设计的。

单任务 vs 多任务
-----------------

- :strong:`单任务` ：一个时刻只能运行一个程序——MS-DOS 就是单任务的，程序退出才回到 shell
- :strong:`多任务` ：同时运行多个进程——几乎所有现代操作系统都支持多任务

多任务又分为:strong:`协作式` （程序主动让出 CPU）和:strong:`抢占式` （操作系统强制切换）。Linux 是抢占式多任务系统，这也是它响应灵敏的原因。

从分类看 Linux
================

回到我们本教程的主角——Linux。它在上述分类中的位置：

.. list-table::
   :header-rows: 1
   :widths: 20 50

   * - 维度
     - Linux 的定位
   * - 应用场景
     - 通用——覆盖桌面、服务器、嵌入式、移动
   * - 内核架构
     - 宏内核（支持模块化动态加载）
   * - 多用户
     - 是
   * - 多任务
     - 抢占式
   * - 实时性
     - 不是原生实时系统（PREEMPT_RT 可接近实时）

Linux 的独到之处在于它的:strong:`模块化设计` ：虽然内核是宏内核，但驱动和功能模块可以在运行时加载和卸载（`insmod` / `rmmod`），这在一定程度上综合了宏内核的性能和微内核的灵活性。
