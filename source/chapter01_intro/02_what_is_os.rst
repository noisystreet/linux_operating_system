======================
什么是操作系统
======================

上一节我们从历史的角度看了操作系统"怎么来的"，现在来回答一个更直接的问题：:strong:`操作系统到底是什么？`

计算机系统可以抽象为四个层次：

.. code-block:: text

   应用程序（浏览器、编辑器、数据库…）
   ────────────────────────────────
   操作系统
   ────────────────────────────────
   硬件（CPU、内存、磁盘、网卡…）

操作系统位于硬件与应用程序之间。从这个位置出发，它的职责可以概括为两句话：

1. :strong:`向下管理硬件` —— 抽象硬件细节，为上层提供统一的接口
2. :strong:`向上提供服务` —— 为应用程序提供运行环境和基础能力

管理硬件：操作系统在做什么？
============================

打开一台 Linux 机器，你可能同时运行着浏览器、终端、音乐播放器、后台编译任务……这背后是操作系统在协调一切。

CPU 管理
--------

你的电脑可能只有 4 个或 8 个物理核心，但你能同时打开几十个应用程序。这不是魔法——操作系统通过:strong:`时分复用` （time-multiplexing）让每个程序轮流使用 CPU，切换速度极快（毫秒级），让你感觉它们在"同时运行"。

这个机制叫做:strong:`进程调度` （scheduling）。操作系统负责决定：下一个轮到谁、运行多久、切换时需要保存和恢复哪些状态。

内存管理
--------

每个程序都以为自己拥有整个地址空间——从 0x00000000 到 0xFFFFFFFF。但实际物理内存是共享的。操作系统通过:strong:`虚拟内存` （virtual memory）给每个进程制造了"独占内存"的假象。CPU 中的 :strong:`MMU` （Memory Management Unit）配合操作系统维护的页表，将虚拟地址翻译成物理地址。

这样做的好处：

- :strong:`隔离` ：进程 A 无法访问进程 B 的内存
- :strong:`安全` ：用户程序无法触碰内核内存
- :strong:`超用` ：物理内存不够时，把暂时不用的数据换到磁盘

存储管理
--------

文件是你和数据的接口，而文件系统是操作系统管理持久化数据的方式。操作系统把磁盘抽象成目录树，你不需要关心数据到底在哪个扇区、哪个磁道。

用户态 vs 内核态
================

操作系统还有一个关键设计：:strong:`特权级分离`。

x86-64 架构提供了四个特权级（ring 0-3）。Linux 只用了两个：

- :strong:`内核态` （kernel mode, ring 0）—— 可以执行任何指令、访问任何硬件
- :strong:`用户态` （user mode, ring 3）—— 只能执行受限指令，不能直接操作硬件

应用程序运行在用户态。当它需要操作系统帮忙——比如读写文件、创建进程——必须通过:strong:`系统调用` （system call）切换到内核态。

.. code-block:: c

   // 用户程序调用 write —— 这背后是系统调用
   write(fd, buf, len);

   // 从用户态陷入内核态，内核完成实际写入
   // 然后返回到用户态

这种隔离机制保证了系统的稳定性：一个应用程序的崩溃不会拖垮整个系统。

操作系统的核心组件
==================

综合来看，一个通用的操作系统至少包含以下模块：

.. list-table::
   :header-rows: 1
   :widths: 20 60

   * - 组件
     - 职责
   * - 进程管理
     - 创建、调度、终止进程；线程管理；同步与通信
   * - 内存管理
     - 虚拟地址空间管理；物理内存分配；页表管理
   * - 文件系统
     - 文件的创建、读写、删除；目录结构；权限控制
   * - 设备驱动
     - 管理硬件设备；提供统一的 I/O 接口
   * - 网络协议栈
     - TCP/IP 协议实现；socket 接口
   * - 系统调用接口
     - 用户态进入内核态的唯一入口
   * - 安全机制
     - 权限检查、Capabilities、LSM

从用户的角度来看，操作系统是一个"看不见的管理者"；从开发者的角度来看，操作系统的本质是:strong:`接口`——一组约定的规范，让程序不必关心底层硬件细节就能使用计算资源。

本书后续的章节将逐一深入这些组件，每一章都会结合 Linux 的实际实现和命令行工具，让你亲手验证操作系统的行为。

在进入细节之前，下一节先看看操作系统有哪些不同的种类——并不是所有的操作系统都像你面前的 Linux 这样通用。

x86-64 特权级与系统调用边界
================================

现代 CPU 用:strong:`特权级` （ring）区分内核态与用户态。x86-64 上 Linux 主要使用 ring 0（内核）和 ring 3（用户程序）：

.. code-block:: text

   ring 3  用户程序（应用程序、shell、库）
              │  syscall / 中断 / 异常
              ▼
   ring 0  内核（进程调度、内存管理、驱动、协议栈）

用户程序:strong:`不能` 直接访问硬件或修改页表；必须通过系统调用请求内核代为操作。这正是上一节「系统调用接口」存在的硬件基础。

操作系统启动流程梗概
==========================

从按下电源到第一个用户进程，可概括为：

.. code-block:: text

   固件(UEFI/BIOS) → Bootloader(GRUB) → 内核(start_kernel)
        → 挂载根文件系统 → init(PID 1, systemd) → 用户登录

第 2 章将沿这条链路逐段展开；届时可用 ``dmesg``、``/boot`` 目录和 ``systemd-analyze`` 对照验证。
